3 ОПИС ВИКОНАНОЇ РОБОТИ
3.1 Загальні відомості про шаблони (патерни) проєктування
     Шаблони проєктування є перевіреними підходами до розв’язання типових задач, які регулярно виникають під час розробки програмного забезпечення. Їх головне призначення полягає не в наданні готового коду, а у формуванні правильної структури системи, що робить її зрозумілішою, гнучкішою та зручнішою для подальшого розвитку. Використання патернів допомагає уникати дублювання, зменшує надмірну залежність між компонентами та полегшує супровід проєкту. 
     Класична класифікація GoF поділяє шаблони на три великі групи: породжувальні, структурні та поведінкові. Породжувальні відповідають за способи створення об’єктів, структурні описують побудову зв’язків між класами й об’єктами, а поведінкові визначають характер їхньої взаємодії. Область застосування патернів досить широка: від невеликих навчальних прикладів до складних програмних систем, де важливо забезпечити розширюваність архітектури без постійного переписування вже реалізованого коду. У межах цієї роботи було розглянуто саме структурний патерн Decorator.
3.2 Опис обраного патерна
• Назва патерна: Decorator;
• Тип шаблона (патерна):  структурний;
• Основне призначення: патерн Decorator призначений для гнучкого розширення поведінки об’єкта без зміни його базового класу. Він дає можливість додавати нові можливості послідовно, через обгортання об’єкта іншими об’єктами, які реалізують той самий інтерфейс;
• Передумови застосування: використання цього патерна є доцільним тоді, коли в системі існує багато варіантів додаткової поведінки, а реалізація через спадкування призводить до надмірного розростання кількості класів. Decorator особливо корисний у випадках, коли функціональність потрібно комбінувати в різних поєднаннях, наприклад логування, шифрування, перевірка доступу, кешування, буферизація або перетворення даних.

• Переваги: 
o патерн забезпечує гнучке розширення функціональності;
o дозволяє дотримуватися принципу відкритості-закритості;
o зменшує залежність від жорсткої ієрархії спадкування;
o дає змогу комбінувати додаткову поведінку в потрібному порядку;
o полегшує розвиток системи без зміни вже існуючих компонентів.
• Недоліки:
o за великої кількості декораторів структура може ускладнюватися;
o важче швидко визначити, які саме обгортки застосовані до об’єкта;
o налагодження може бути складнішим через розподіл логіки між кількома невеликими класами;
o у простих системах використання Decorator інколи є надмірним.
3.3 Структура шаблона (патерна)
     До складу патерна Decorator входить кілька основних структурних частин. Першою з них є Component, тобто спільний інтерфейс або абстракція, яку реалізують як базовий компонент, так і всі декоратори. Саме завдяки цьому клієнтський код може працювати з ними однаково. Другою складовою є ConcreteComponent, тобто базовий об’єкт, який виконує основну дію. Третьою є Decorator, який також реалізує інтерфейс Component, але додатково містить посилання на інший об’єкт типу Component. Четвертою складовою є ConcreteDecorator, який додає конкретну нову поведінку до або після виклику базового методу.
• Опис структурних частин:
o Component – спільний інтерфейс для базового компонента й декораторів;
o ConcreteComponent – клас, що реалізує основну поведінку;
o Decorator – базова обгортка, яка зберігає посилання на Component;
o ConcreteDecorator – конкретний декоратор, який додає нові можливості.
• Опис взаємодії об’єктів
     Принцип взаємодії полягає в тому, що клієнт працює не безпосередньо з конкретним класом, а зі спільним типом Component. Базовий об’єкт може бути послідовно обгорнутий кількома декораторами. Кожен декоратор приймає компонент, зберігає посилання на нього та під час виклику методу передає основну роботу вкладеному об’єкту, додаючи власну дію до або після основної операції. У результаті формується ланцюг поведінки, у якому кожне нове розширення не змінює початковий механізм, а лише доповнює його. Такий підхід був показаний у доповіді на прикладі системи повідомлень і в роботі з файлами та потоками даних.
• UML-діаграми з поясненнями

Рисунок 1 – UML-діаграма Decorator
     UML-діаграма класів для патерна Decorator показує інтерфейс Component, клас ConcreteComponent, базовий клас Decorator із посиланням на Component та конкретні декоратори, які наслідують Decorator. Пояснення до діаграми полягає в тому, що всі учасники мають спільний інтерфейс, а декоратори розширюють функціональність через композицію, а не через створення великої кількості підкласів. Це і є головною ідеєю шаблона.
3.4 Реалізація шаблона (патерна). ДОДАТОК В
• Опис реалізації мовою програмування: JavaScript
     У межах практичної частини патерн Decorator було реалізовано мовою JavaScript на прикладі системи надсилання повідомлень. Базовий клас Message виконує лише одну просту дію – надсилання повідомлення. Додаткова поведінка, така як логування, шифрування, перевірка доступу та кешування, була винесена в окремі декоратори. Кожен із них розширює базовий функціонал, не змінюючи сам клас Message. Такий підхід дозволяє не створювати окремі класи для всіх можливих поєднань функцій, а формувати потрібну поведінку шляхом послідовного обгортання базового об’єкта декораторами.
• Основні етапи:
o Створення базових інтерфейсів
     У ролі базового компонента використано клас Message, який містить метод send() і виконує основну дію, тобто надсилання повідомлення. Також створено клас MessageDecorator, який зберігає посилання на об’єкт повідомлення та передає виклик методу далі. Саме він виступає основою для всіх конкретних декораторів.
o Реалізація конкретних класів
     Для розширення функціональності базового повідомлення реалізовано декоратори LoggingDecorator, EncryptionDecorator, AccessDecorator і CacheDecorator. LoggingDecorator додає запис у журнал перед надсиланням повідомлення. EncryptionDecorator змінює текст повідомлення, імітуючи його шифрування. AccessDecorator перевіряє, чи дозволено виконувати надсилання.
CacheDecorator зберігає вже надіслані повідомлення та не допускає повторного надсилання однакового тексту.
o Демонстрація використання
     Роботу патерна показано на кількох прикладах. У першому випадку до базового повідомлення додається лише логування. У другому поєднуються логування та шифрування. У третьому прикладі використано послідовне обгортання базового об’єкта декораторами кешування, перевірки доступу, логування та шифрування. Це показує, що нова поведінка формується гнучко, без створення окремих класів для кожної комбінації можливостей.

Приклади програмного коду та запити до ШІ
     Промт 1, з використанням патерна Decorator: Напиши компактний приклад на JavaScript, який демонструє використання патерна проєктування Decorator на прикладі системи надсилання повідомлень. Створи базовий клас Message з методом send(t), який виводить у консоль `Надсилання повідомлення: ${t}`. Створи базовий декоратор MessageDecorator, який приймає об’єкт message через конструктор і передає виклик методу send(t) вкладеному об’єкту. Потім реалізуй декоратори LoggingDecorator, EncryptionDecorator, AccessDecorator і CacheDecorator. LoggingDecorator має виводити `[LOG] Спроба надсилання повідомлення`, EncryptionDecorator має передавати далі текст у вигляді `***${t}***`, AccessDecorator має перевіряти умову `!true` і в разі істинності виводити `[ACCESS] Доступ заборонено`, CacheDecorator має використовувати Set і виводити `[CACHE] Повідомлення вже було надіслане раніше`, якщо текст уже надсилали. Наприкінці покажи використання через `const msg1 = new LoggingDecorator(new Message()); msg1.send("Привіт");`, `const msg2 = new LoggingDecorator(new EncryptionDecorator(new Message())); msg2.send("Секретне повідомлення");`, `const msg3 = new CacheDecorator(new AccessDecorator(new LoggingDecorator(new EncryptionDecorator(new Message())))); msg3.send("Важливий текст"); msg3.send("Важливий текст");`. Виведи тільки готовий код без пояснень. Код має бути компактний, без зайвих переносів, але читабельний.
     Результат промту 1:
1 class Message {
2 send(t) { console.log(`Надсилання повідомлення: ${t}`); }
3 }
4 class MessageDecorator {
5 constructor(message) { this.message = message; }
6 send(t) { this.message.send(t); }
7 }
8 class LoggingDecorator extends MessageDecorator {
9 send(t) { console.log("[LOG] Спроба надсилання повідомлення"); super.send(t); }
10 }
11 class EncryptionDecorator extends MessageDecorator {
12 send(t) { super.send(`***${t}***`); }
13 }
14 class AccessDecorator extends MessageDecorator {
15 send(t) {
16 if (!true) { console.log("[ACCESS] Доступ заборонено"); return; }
17 super.send(t);
18 }
19 }
20 class CacheDecorator extends MessageDecorator {
21 constructor(message) { super(message); this.cache = new Set(); }
22 send(t) {
23 if (this.cache.has(t)) { console.log("[CACHE] Повідомлення вже було надіслане раніше"); return; }
24 this.cache.add(t); super.send(t);
25 }
26 }
27 const msg1 = new LoggingDecorator(new Message());
28 msg1.send("Привіт");

29 const msg2 = new LoggingDecorator(new EncryptionDecorator(new Message()));
30 msg2.send("Секретне повідомлення");

31 const msg3 = new CacheDecorator(new AccessDecorator(new LoggingDecorator(new EncryptionDecorator(new Message()))));
32 msg3.send("Важливий текст");
33 msg3.send("Важливий текст");

     Промт 2, без використання патерна: Напиши компактний приклад на JavaScript, який показує ту саму систему надсилання повідомлень, але без використання патерна Decorator, лише через спадкування. Створи базовий клас Message з методом send(t), який виводить у консоль `Надсилання повідомлення: ${t}`. Потім створи класи LoggedMessage, EncryptedMessage, AccessCheckedMessage, CachedMessage, LoggedEncryptedMessage, LoggedEncryptedAccessCheckedMessage і LoggedEncryptedAccessCheckedCachedMessage. У цих класах вручну поєднай логування, шифрування, перевірку доступу та кешування так, щоб було видно проблему розростання кількості класів через комбінації поведінки. Наприкінці покажи використання через `const msg1 = new LoggedMessage(); msg1.send("Привіт");`, `const msg2 = new LoggedEncryptedMessage(); msg2.send("Секретне повідомлення");`, `const msg3 = new LoggedEncryptedAccessCheckedCachedMessage(); msg3.send("Важливий текст"); msg3.send("Важливий текст");`. Виведи тільки готовий код без пояснень. Код має бути компактний, без зайвих переносів, але читабельний.

1 class Message {
2 send(t) { console.log(`Надсилання повідомлення: ${t}`); }
3 }
4 class LoggedMessage extends Message {
5 send(t) { console.log("Спроба надсилання повідомлення"); super.send(t); }
6 }
7 class EncryptedMessage extends Message {
8 send(t) { super.send(`***${t}***`); }
9 }
10 class AccessCheckedMessage extends Message {
11 send(t) {
12 if (!true) return console.log("[ACCESS] Доступ заборонено");
13 super.send(t);
14 }
15 }
16 class CachedMessage extends Message {
17 constructor() { super(); this.cache = new Set(); }
18 send(t) {
19 if (this.cache.has(t)) return console.log("[CACHE] Повідомлення вже було надіслане раніше");
20 this.cache.add(t); super.send(t);
21 }
22 }
23 class LoggedEncryptedMessage extends Message {
24 send(t) {
25 console.log("[LOG] Спроба надсилання повідомлення");
26 super.send(`***${t}***`);
27 }
28 }
29 class LoggedEncryptedAccessCheckedMessage extends Message {
30 send(t) {
31 if (!true) return console.log("[ACCESS] Доступ заборонено");
32 console.log("[LOG] Спроба надсилання повідомлення");
33 super.send(`***${t}***`);
